Eventos Objeto

Então, o que são eventos objetos? Basicamente, estes são momentos discretos no loop do jogo onde as coisas são feitas para acontecer com base no que você programou para eles. O GameMaker Studio 2 trabalha com ciclos desses eventos - a partir do momento em que um sala é iniciada até o momento em que é concluída, há um loop de jogo correndo onde a cada passo uma série de eventos são executados ou verificados, e você pode escolher colocar código ou DnD™ações em seus objetos que respondam a esses eventos (um passo é um momento no tempo de jogo, governado pela configuração de velocidade da sala, e também pode ser chamado de frame) .

Esta é a lista de todos os eventos básicos e categorias de eventos aos quais um objeto pode responder, e dentro de cada categoria são sub eventos para refinar ainda mais os comportamentos. Por exemplo, se você clicar na categoria de eventos daKey Press , você obter mais uma janela que aparece com os eventos que permitem selecionar a qual chave o objeto deve responder. Uma vez selecionado seu evento, a janela de código (ou a janela de espaço de trabalho DnD™) será acorrentada a ele e se abrirá à direita. Agora você pode editar o código para dar a seu objeto um comportamento ou reação específica a esse evento.

Você pode clicar com o botão direito do mouse em qualquer evento que tenha sido adicionado a um objeto para obter as seguintes opções de menu:

Estas opções são:

  • Adicionar Evento - Adicionar um novo evento a partir da lista de eventos.
  • Cortar / Copiar / Colar Eventos - Cortar, copiar ou colar o evento selecionado. Você também pode usar os atalhos de teclado padrão: / + X, / + C, e / + V.
  • Evento Duplicado - Duplicar o evento escolhido. Isto levará o conteúdo do evento escolhido e os duplicará em outro evento que você especificar.
  • Change Event - Mude o evento. Isto o levará a escolher outra categoria de evento e uma vez selecionado, o conteúdo do evento atual será alterado para o novo evento (removendo o evento original no processo).
  • Convert To Drag and Drop / Code - Os eventos do objeto serão criados usando o método escolhido quando você criou o projeto, seja usando código ou usando as ações Drag e Drop™. Com esta opção você pode converter de um para o outro, independentemente do tipo de projeto.
  • Eliminar evento - Eliminar o evento (ou eventos).

Ao remover eventos, você pode usar + para selecionar vários eventos Cada objeto que você cria tem sua própria lista discreta de eventos que são adicionados a ele a partir do Editor de Objetos. Estes eventos se encaixam em duas categorias:

  • aqueles que executam cada passo do jogo
  • aqueles que são "acionados" por um evento de jogo, como a instância que atinge a borda da sala ou um teclado ou um apertar de mouse.

A lista completa dos eventos é dada abaixo:

Criar

Este evento acontece quando uma instância do objeto é criada pela primeira vez, e é a primeira coisa que acontece dentro de uma instância colocada na sala através do Editor de Sala quando uma sala é entrada. Isto significa que este evento é o lugar ideal para inicializar variáveis, iniciar linhas de tempo, definir Caminhos, etc... e fazer qualquer outra coisa que geralmente só precisa de ser feito uma vez ou apenas quando uma instância é criada na sala pela primeira vez. Se seu objeto tiver alguma variável de objeto ou variável de instância adicionada no Editor de Objetos ou no Editor de Sala, então estas variáveis serão iniciados primeiro e, em seguida, será realizado o Create Event.

Lembre-se de que você pode modificar qualquer coisa que tenha configurado no Create Event a partir do Código de Criação de Instância no Editor de Sala, pois este é executado diretamente após a criação do evento para a instância e pode ser usado para criar Variáveis de Instância ou para substituir quaisquer variáveis adicionadas como Variáveis de Objeto ou no próprio Evento Criar.

Destruir

Este evento é o evento a ser executado quando uma instância é destruída. É freqüentemente negligenciado quando se adicionam comportamentos a objetos, mas pode ser muito útil, por exemplo, criando efeitos de explosão ou de partículas quando um inimigo é morto, ou para re-despertar uma nova instância do objeto em outra parte da sala, ou mesmo para somar pontos a uma pontuação.

Limpeza

Este evento será chamado após qualquer evento que remova uma instância do objeto da sala. Portanto, ele será acionado se:

  • o exemplo é destruído
  • a sala termina
  • o jogo termina

É projetado para você usar para "limpar" quaisquer recursos dinâmicos que você possa ter em seu jogo (como superfícies, estruturas de dados, etc...) ou para executar qualquer tarefa que você precise executar uma vez quando a instância for removida do jogo de qualquer forma.

Note que este evento será chamado imediatamente após o evento que o desencadeou, mas a instância não será realmente removida do jogo até o final do evento atual. Por exemplo, se você ligar para instance_destroy() no Step Event, então o Destroy Event será chamado, então o Clean Up Event, e então o resto do Step Event terminará de funcionar. Isto significa que qualquer código que você tenha após a chamada para instance_destroy() ainda será executado e será uma causa potencial para erros se você tiver limpo uma estrutura de dados ou algum outro recurso que o código requeira, portanto é preciso ter cuidado ao usar este evento.

Alarme

A categoria de alarme é dividida em 12 eventos, um para cada um dos alarmes possíveis que podem ser definidos em uma instância. Assim, quando você clica na categoria Adicionar alarme , esta janela lhe é apresentada:

Aqui você seleciona o alarme que deseja criar e, uma vez feito isso, você verá que ele foi adicionado ao evento permitindo adicionar código a ela como de costume. Mas o que é um alarme? Bem, é um evento especial que não faz nada a menos que o alarme tenha sido previamente ajustado, e então ele esperará até que o alarme tenha contado até 0 antes de executar as ações ou código que você adicionou a ele.

Uma vez que o alarme tenha atingido 0 e executado o código, ele então contará até -1, onde permanecerá até ser ajustado novamente (ou seja, você pode verificar o valor de um alarme para ver se ele é maior que -1, o que lhe dirá se ele está funcionando ou não). Então, digamos você definir alarm[0] no evento de criação do objeto para 30, isto significa que o GameMaker Studio 2 contará 30 passos de jogo antes de executar as ações ou código que são colocados no evento alarm[0]. Note que definir um alarme para 0 não irá executar o código do alarme, pois o evento é acionado, mas o alarme é definido para -1 imediatamente, então o código é pulado. Se você precisar de um alarme para executar o próximo passo, então você deve configurá-lo para 1.

Isto pode ser muito útil, pois permite que você ponha as coisas em movimento em momentos precisos, e você pode até mesmo fazer com que elas se repitam, pois não há nada que o impeça de dar um alarme em seu próprio evento. Imagine que você tem um monstro e quer que ele vire à direita a cada três segundos... bem, você colocaria um alarme em seu evento de criação na velocidade da sala * 3 (se a velocidade da sala for 30, são 30 passos por segundo, então multiplique isso por 3 e você terá 3 segundos!) e então no evento de alarme você teria o código ou ação para definir sua direção, assim como a ação (ou código) para ajustar seu alarme à velocidade da sala * 3 novamente. Desta forma, você pode configurar laços de jogo simples onde as coisas só acontecem em intervalos específicos.

Vale notar que um alarme sem ações ou código nele contido não contará para baixo. Entretanto, mesmo com apenas um comentário e sem código ou ações, então o alarme continuará a contagem regressiva e pode ser ajustado e verificado como você normalmente faria.

Etapa

OGameMaker Studio 2 divide o tempo em etapas com a velocidade da sala definindo quantas dessas etapas devem ser executadas por segundo (uma etapa também pode ser chamada de frame). Um único passo, é basicamente o loop que corre constantemente com todos os eventos sendo verificados e acionados conforme necessário enquanto o jogo decorre, de modo que, como você pode imaginar, o Evento Escalonado é um evento que é verificado em cada uma das etapas do jogo enquanto a instância existe.

O evento da etapa é na verdade composto de três subeventos que são descritos abaixo:

Para a maioria das coisas, o evento da etapa padrão será bom de usar, mas às vezes você quer um pouco mais de controle sobre o que o código funciona e em a que horas, para que você tenha acesso aos eventos dos passos de Início e Fim . Todos os três são verificados a cada etapa, mas seu pedido nunca irá variar, mesmo que futuras atualizações do motor do GameMaker Studio 2 mudem outros o que significa que este é o único método confiável de garantir que algo sempre aconteça antes de algo mais.

Para que o evento da etapa pode ser usado? Bem, pode ser usado para ações ou código que precisa ser executado continuamente. Por exemplo, se um objeto deve seguir outro, aqui você pode adaptar a direção do movimento em direção ao objeto que estamos seguindo para mantê-lo em movimento sem problemas. Mas tenha cuidado com este evento, e não coloque muitas ações complicadas no evento do passo de objetos, especialmente se você estiver planejando ter muitas instâncias do objeto em sua sala de jogos, pois isto pode retardar o jogo. Muitas coisas podem ser colocadas em alarmes, ou definidas para acionar usando alguns dos Outros eventos, em vez de acontecer o tempo todo.

Colisão

Obviamente, ao fazer um jogo, é muito importante que você saiba quando duas (ou mais) instâncias de um objeto colidiram, e para isso temos o Evento de Colisão. Este é um evento que você coloca em um objeto e depois especifica contra qual outro objeto você deve estar verificando se há colisões.

Quando você não tem a física ligada, essas colisões serão calculadas com base na máscara dos dois objetos (a máscara é definida dentro das propriedades do duende, ou pode ser atribuída independentemente nas propriedades do objeto) e se elas se sobrepõem ou não. Observe que se uma ou outra instância na colisão não tiver uma máscara atribuída (ou a máscara de duende estiver ajustada a nada), mesmo que ela esteja desenhando algo, nenhuma colisão será detectada.

Se você tiver Física , então a colisão será baseada no tipo de forma de colisão(Fixture) que você definiu para o objeto em suas propriedades físicas, assim como sua reação à colisão. Este significa que você pode não precisar de nenhum código para lidar com a colisão, mas este evento ainda precisará ter pelo menos um comentário nele para que as colisões sejam detectadas.

Finalmente, deve-se observar que todas as colisões serão calculadas uma vez por etapa do jogo antes do evento da colisão ser acionado, de modo que, quando o evento da colisão ocorrer, todas as colisões já terão sido calculadas e pré-atribuídas. Isto significa que se você criar uma instância neste evento e depois tentar verificar uma colisão com ela, a colisão não será detectada ou resolvida até a próxima iteração do loop do jogo.

Teclado, Prensa de Teclado, Liberação do teclado

Deixar o jogador controlar os diferentes aspectos de seu jogo é muito importante, e para isso o GameMaker Studio 2 fornece uma lista muito abrangente de eventos de teclado que podem ser usados em qualquer uma das três principais categorias de teclado. Para o categoria geral do Teclado, ele é acionado continuamente a cada passo, enquanto a tecla selecionada for pressionada para baixo, enquanto os eventos da categoria Imprensa e Liberação só serão acionados uma vez quando a tecla for inicialmente pressionada ou liberada.

Deve-se observar que os eventos de teclado são realmente acionados em todas as instâncias ativas em uma sala sempre que uma tecla é usada, mas somente aqueles que têm um evento definido para aquela tecla específica responderão e você pode criar múltiplos eventos de teclado em qualquer objeto e as instâncias desse objeto responderão a todos eles enquanto o jogo estiver em andamento.

Ao adicionar qualquer evento de teclado a um objeto, será apresentado a você o menu de subeventos de teclado onde você pode especificar a chave que você deve estar verificando:

A maioria deles são bastante óbvios, mas vamos apenas percorrer as seções brevemente - no topo temos as setas do teclado, seguido pelas teclas modificadoras mais usadas, depois o resto do teclado (dividido em outras subseções para que você possa obter a chave exata necessária como ou ) e finalmente dois subeventos muito especiais, Sem Chave e Qualquer Chave. Como seus nomes implicam, estes são sub eventos que verificam quando nenhuma tecla é pressionada ou para quando qualquer tecla é pressionada. Favor observar que as teclas do teclado numérico só produzem os eventos correspondentes quando o Bloqueio Numérico está ativado.

Os eventos de Imprensa e Lançamento para o teclado são quase exatamente os mesmos que o evento normal do teclado, exceto que, em vez de serem acionados continuamente, eles são acionados apenas uma vez. Quando o teclado registra pela primeira vez que uma tecla foi pressionada irá gerar um evento de Teclado pressionado (assim como um evento regular de Teclado), e a primeira vez depois disso onde uma tecla não está mais sendo detectada como pressionada irá acionar um único evento de Liberação de Teclado.

Mouse

A categoria Mouse é separada em uma série de eventos que podem ser selecionados para lhe dar um controle mais preciso sobre o que está acontecendo em seu jogo. Aqui você pode ver exatamente o que são esses eventos:

A esquerda , direita e meio eventos com botões (sejam normais, pressionados ou liberados) todos trabalho sobre a máscara da instância que tem o evento. Isto significa que o GameMaker Studio 2 verificará a posição do mouse na sala quando esses botões forem usados contra as máscaras de colisão das instâncias que têm um evento com o mouse. Se há uma "colisão" com a caixa de delimitação da instância , então o evento será acionado, então certifique-se de que qualquer instância com esses eventos tenha um duende com uma colisão válida máscara ou que o objeto tem um duende de máscara selecionado nas propriedades do objeto. Como seus nomes implicam, estes eventos serão acionados uma vez quando o botão do mouse escolhido for pressionado ou liberado pela primeira vez, ou continuamente cada passo enquanto o botão estiver mantido.

Os eventos de entrada e saída do mouse também são similares aos eventos de botões, pois eles também dependem da máscara da instância para trabalhar, mas desta vez são acionados quando o mouse "entra" (toca) pela primeira vez na instância ou quando o mouse "sai". (pára de tocar) a instância. Estes eventos não são contínuos, porém, e são acionados apenas uma vez para cada vez que o mouse entra ou sai do objeto - portanto, são um método ideal para criar, por exemplo, botões que precisam mudar conforme o mouse paira sobre eles antes de voltar ao normal quando o mouse é removido.

Finalmente temos outra seção para os eventos do mouse que é chamada de Global Mouse. Neste sub-menu você encontrará uma seleção de eventos que são para registrar eventos do mouse em instâncias mesmo quando o mouse não está sobre eles ou mesmo perto deles. Estes são eventos que são gerados para todas as instâncias e se houver ações ou código definido para o evento especificado então ele será executado, independentemente da posição do mouse dentro da sala de jogo.

Observe que em dispositivos móveis ou de tela sensível ao toque o botão esquerdo do mouse também pode ser usado para verificar se há uma aba com o dedo em uma tela sensível ao toque, e o botão direito do mouse é acionado por um toque duplo na tela (este comportamento pode ser alterado usando código).

Gestos

Este evento é aquele que será acionado pelo usuário tocando a tela (no celular) ou clicando e movendo o mouse (em todas as outras plataformas). Estes eventos são semelhantes aos eventos do mouse, na medida em que você tem versões regulares e versões globais. As versões regulares destes eventos só serão acionadas quando os toques ocorrerem em uma instância que tenha um sprite (ou uma máscara) e os toques ocorrerem dentro de sua caixa delimitadora. A versão global No entanto, as versões destes eventos serão acionadas pelo usuário tocando em qualquer lugar na tela.

Os eventos gestuais detectam o seguinte:

  • taps - quando um usuário clica/toca e solta rapidamente
  • arrasta - quando um usuário toca/clica e o mantém, então move seu dedo/cursor
  • flicks - quando o usuário se move e solta um toque/clique em um movimento
  • pitadas - quando o usuário tem dois dedos na tela e depois os move juntos/apart
  • gira - quando o usuário tem dois dedos na tela e depois os gira em torno de um ponto

Os diferentes eventos sempre conterão um mapa DS chamado "event_data", que conterá um número de pares chave/valor com dados sobre a posição de toque/clique e movimento. Para detalhes completos sobre todos os subeventos disponíveis e como eles funcionam, consulte a seção seguinte:

Sorteio

Esta categoria de eventos é a que rege o que você vê na tela quando dirige seu jogo, e está dividida em vários eventos discretos:

Como você pode ver, a categoria de eventos de sorteio tem vários tipos diferentes de eventos. O início, o fim do sorteio e o sorteio são os eventos de sorteio "padrão" que você provavelmente utilizará mais. Por padrão o evento de sorteio principal é sempre chamado para cada instância, independentemente de ter ou não um duende, embora se você marcar a instância como invisível, o evento não será acionado (portanto, tenha isto em mente se você tiver alguma lógica de jogo no evento do sorteio de um objeto invisível, pois ele não funcionará). O evento de sorteio principal é também onde o GameMaker Studio 2 desenha a instância sprite quando não há código nem ações no evento (ou seja: você não adicionou na lista de eventos para o objeto). O desenho padrão utiliza o sprite associado à instância e desenhará aquele com qualquer transformação definida em código ou ações aplicadas.

Os eventos padrão do sorteio são sorteados antes dos eventos da GUI e entre os eventos de Pré-Sorteio e Pós-Sorteio , o que significa que tudo o que é sorteado neste evento é sorteado abaixo disso do evento de Sorteio GUI , independentemente da camada (ou seja: qualquer coisa desenhada no evento de Sorteio GUI será sempre desenhada sobre qualquer coisa desenhada no evento de Sorteio normal, independentemente da ordem da camada).
